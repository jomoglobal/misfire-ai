# Compliance Note
*MisfireAI · May 2026*

---

## What Data This System Touches

MisfireAI processes vehicle diagnostic data captured directly from the owner's OBD2 port:

- **OBD2 session data** — Mode 01 PIDs (RPM, fuel trims, coolant temp, MAF/MAP, O2 sensors, timing advance, catalyst temp, throttle, load)
- **DTCs** — active and pending fault codes
- **VIN** — used for vehicle identification, NHTSA decode, and TSB/recall lookup. Hashed before any external storage.
- **Health scores** — computed 0–1 per system, derived from session data
- **Repair briefs** — plain-language diagnostic summaries generated by the agent

All data originates from hardware the vehicle owner controls. No data is scraped, purchased, or sourced from third parties. NHTSA API calls (VIN decode, TSB lookup) are outbound only — no session data is transmitted to NHTSA.

## Data Collection Policy

MisfireAI operates on a two-tier collection model:

### Free / Demo Tier — Default Collection
Users of the free product accept data collection as a condition of use, disclosed at first launch. Anonymized session data (VIN hashed, no GPS, no owner identity) is collected to:
- Improve diagnostic accuracy across makes and models
- Build comparative baselines (e.g., "your Tundra's LTFT is +4% — median for 1GR-FE engines at this mileage is +1.5%")
- Train and validate the scoring models

This is the standard model used by every major OBD2 application. The value exchange is explicit: free diagnostics in exchange for contributing to the shared dataset.

### Paid Tier — Opt-In with Stated Benefit
Paying customers are given a clear choice at onboarding:

> *"Share anonymized session data to contribute to fleet intelligence and unlock comparative benchmarks against similar vehicles and engine families."*

Opting in is framed as a benefit, not a requirement. Opting out has no impact on paid feature access.

### What Is Never Collected (Regardless of Tier)
- GPS coordinates or precise location
- Owner name, contact information, or identity
- Financial information
- Raw VIN (always hashed before leaving the device)
- Data from any source other than the owner's own vehicle hardware

## What Requires Human Approval

The pipeline produces outputs at four severity levels. Only high-stakes, irreversible actions are gated.

| Action | Stakes | Reversible | Gate |
|---|:---:|:---:|---|
| Session report generated | Low | — | None — auto-logged |
| Anomaly classified (Tier 1–4) | Medium | Yes | Auto-logged with reasoning and confidence |
| Health score computed | Medium | Yes | Auto-logged |
| **Repair brief** | **High** | **No** | **Owner approval required before any action** |
| DTC clear (Mode 04) | High | No | **Blocked — permanently out of scope** |
| Data shared externally | High | No | **Blocked without explicit owner consent** |
| Writing to vehicle ECU | High | No | **Blocked — outside technical scope** |

### Approval Flow — Repair Brief
When the agent generates a repair brief, the pipeline halts and delivers the brief to the owner via one of three channels (configurable):

- **Email** — brief delivered with approve / reject links
- **Telegram** — bot message with inline approve / reject buttons
- **Slack** — channel message with interactive approve / reject actions

The owner reviews: vehicle, flagged systems, health scores, recommended action, and estimated urgency tier. They approve or reject. The decision — and the timestamp — is logged regardless of outcome. No repair brief is acted upon, forwarded, or stored in any shared system without an explicit approval event on record.

## What Gets Logged

Every pipeline run produces a full audit trail in Phoenix/Arize OTel tracing:

| Event | What Is Logged |
|---|---|
| `ingest_file` | Source format, row count, PID count, warnings, timestamp |
| `decode_vin` | VIN (hashed), decoded make/model/year, NHTSA response |
| `lookup_tsb` | VIN (hashed), recall count, complaint count |
| `score_vehicle_health` | Per-system scores, scoring method used, data gaps noted |
| `classify_anomaly` | Tier assigned, PIDs flagged, reasoning, operating context |
| `generate_repair_brief` | Brief content, delivery channel, owner decision, timestamp |
| HITL gate — approved | Approver, timestamp, brief hash |
| HITL gate — rejected | Rejector, timestamp, rejection logged |

Traces are immutable once written. No log entry is deleted or modified after the fact.

## Hard Nos

These actions are permanently out of scope and not present anywhere in the codebase:

- **No Mode 04 (DTC clear)** — clearing codes destroys the diagnostic record and masks developing faults. This will never be automated.
- **No ECU writes** — MisfireAI is read-only from the vehicle. No calibration, no parameter writes, no reflashing.
- **No external data sharing without a logged approval event** — data does not leave the local pipeline without an explicit owner action on record.
- **No selling or monetizing individual vehicle data** — fleet analytics use only anonymized, aggregated, hashed session summaries.
- **No third-party API calls that transmit session data** — NHTSA lookups use VIN only. No session PIDs or health scores are sent to any external service.

## Data Lifecycle

```
Capture (local hardware)
    ↓
Normalize + Score (local pipeline)
    ↓
Owner reviews + approves repair brief (HITL gate)
    ↓
Free tier: anonymized session summary → fleet dataset (default, disclosed)
Paid tier: same, only if opted in
    ↓
Fleet dataset: VIN hashed, no GPS, no identity — aggregated to session-level
summaries after 90 days (raw rows not retained indefinitely)
```

Local session files remain on the owner's machine. The owner can delete them at any time. MisfireAI does not maintain a remote copy of raw session data without explicit opt-in.
